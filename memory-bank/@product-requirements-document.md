# Growth Insight System - 个人成长洞察系统 PRD

## 1. 产品定位

Growth Insight System 是一个以真实生活数据为基础的个人成长洞察系统。

它不是单纯的待办工具、日记工具或习惯打卡工具，而是一个围绕“记录、统计、复盘、建议、调整”的人生管理工作台。

核心目标：

```text
手动记录日常数据
  ↓
程序自动统计和可视化
  ↓
固定时间调用 AI 做复盘分析
  ↓
生成生活模式洞察和行动建议
  ↓
反过来指导下一阶段计划
```

第一版先服务 Faye 自己，优先做出稳定、真实、低成本的自用闭环。未来可以扩展为公开上线版本。

产品界面方向：

- 做成安静、清晰、实用的 dashboard。
- 不做营销式首页。
- 未登录用户可以浏览产品界面、基础介绍和一套展示数据。
- 用户尝试填写任务、习惯、日程、随手记或生成复盘时，提示注册或登录。

## 2. 用户与问题

### 2.1 第一阶段用户

- Faye，应用心理专业背景，正在转行 AI 产品。
- 希望把日程、任务、习惯、情绪、日记、灵感、纪念日等生活数据统一管理。
- 希望系统能帮助看见生活模式，而不是只做记录。

### 2.2 当前痛点

- 现有 Coze skill 可以记录信息，但使用体验偏脚本化，不像一个完整产品。
- 每天的任务、习惯、日程、情绪、灵感分散在不同入口。
- 现有记录方式更偏临时存放数据，不适合承载完整的人生管理工作台。
- 生活复盘需要 AI，但不希望所有操作都调用 AI，成本和隐私都要可控。
- 学习笔记已经在 Obsidian，不希望第一版重复建设完整知识库。

## 3. 产品原则

1. **手动优先，AI 补位**
   - 任务、习惯、日程等快速操作由用户手动完成。
   - AI 不替代简单录入，只在复盘、归纳、建议时介入。

2. **程序能算的，不交给 AI**
   - 完成率、连续天数、趋势、热力图、分类分布等由程序完成。
   - AI 负责解释和建议，不负责基础统计。

3. **先自用，架构预留产品化**
   - 第一版功能服务 Faye 自己。
   - 数据库、认证、AI 接入方式按未来公开上线预留。

4. **真实生活数据驱动**
   - 系统价值来自长期记录形成的生活轨迹。
   - 可视化和 AI 复盘都必须围绕真实行为、情绪和事件。

5. **低成本和隐私可控**
   - AI 结果缓存，不重复生成。
   - AI 输入优先使用统计结果、摘要和少量必要片段。
   - 支持切换低成本或国内 AI API。

## 4. MVP 功能范围

本文的 MVP 指完整第一版产品边界。当前基础功能第一轮先实现每日工作台、基础记录、基础图表和每日 AI 复盘；周复盘、月度复盘、纪念日、礼物记录和场景工具箱保留在第一版产品方向中，但不进入第一轮开发。

### 4.1 每日工作台

每日工作台是第一版最高优先级入口，但当前定位已经从“铺开所有列表的工作台”收敛为“每日入口页”。

当前打开系统时默认进入每日工作台。页面优先展示：

- 今日概览
  - 任务完成率
  - 习惯打卡
  - 今日日程数量
  - 随手记录数量
  - 四张概览卡可跳转到清单页或人生页对应列表

- 晚间总结入口
  - 跳转复盘界面
  - 在复盘界面查看程序统计、周/月复盘和 AI 复盘入口

具体列表和维护入口分散到更清晰的主导航中：

- 清单页：任务、日程、习惯、灵感
- 人生页：事件、纪念日、礼物
- AI 页：聊天式快捷创建入口
- 复盘页：数据概览、周复盘、月复盘、导出和问题拆解入口

`/daily?view=...` 旧入口仍保留，用于清单页“新增”按钮打开对应创建表单。

功能：

- 今日任务
  - 创建任务
  - 标记完成
  - 设置分类：学习、工作、生活、健康、关系、其他
  - 设置状态：未开始、进行中、已完成、延期
  - 延期任务需要标注为延期，并允许同步修改到延期日期
  - 记录任务复盘

- 习惯打卡
  - 创建习惯
  - 每日打卡
  - 查看连续天数和累计次数
  - 记录中断或复盘原因

- 今日日程
  - 手动记录当天固定事项
  - 支持时间、标题、分类
  - 第一版不做复杂日历同步

- 随手记录
  - 记录事件和灵感
  - 事件：类似片段式日记，用于记录每天比较有意义的事情
  - 灵感：想做但当下还没有安排做的事情，后续可以转化为任务，也可以搁置或放弃
  - 情绪不是独立记录类型，而是在记录事件时手动选择
  - 第一版预设情绪标签：平静、开心、满足、期待、兴奋、焦虑、疲惫、低落、委屈、生气、压力、混乱、孤独、感激
  - 支持标签：情绪、场景
  - 支持简短日记式输入

- 晚间总结入口
  - 每天固定时间或手动点击生成今日复盘
  - 展示今日统计、关键事件、AI 总结和明日建议

### 4.2 成长记录

成长记录用于长期沉淀。

记录类型：

- 任务完成记录
- 习惯打卡记录
- 日程记录
- 事件记录
- 灵感记录
- 纪念日
- 礼物记录
- 每日复盘
- 每周复盘
- 月度复盘

核心目标不是存档，而是为后续生活模式洞察提供数据。

### 4.3 可视化洞察

第一版可视化尽量用程序完成。

基础图表：

- 今日任务完成率
- 本周任务完成率
- 任务分类分布
- 延期任务数量
- 习惯连续天数
- 习惯打卡日历
- 情绪趋势图
- 情绪热力图
- 记录数量趋势
- 周/月成长趋势

洞察问题：

- 哪类任务最容易拖延？
- 哪些习惯最稳定？
- 哪些情绪最近出现频率最高？
- 哪些场景最容易带来消耗？
- 哪些日子记录密度最高或最低？

### 4.4 AI 复盘

AI 只在固定复盘节点介入。

第一版包含：

- 每日复盘
  - 今日整体状态
  - 任务完成情况
  - 习惯执行情况
  - 情绪和关键事件
  - 明日行动建议

- 每周复盘
  - 本周生活模式
  - 消耗来源
  - 正向积累
  - 反复问题
  - 下周 1-3 个行动建议

- 月度复盘
  - 月度主题
  - 成长变化
  - 长期问题
  - 下月重点方向

基础功能第一轮只实现每日复盘。首版部署允许先不上 AI，只展示程序统计、基础图表和每日程序复盘摘要；后续补齐 AI 环境变量后，再开放 AI 解释和建议。每周复盘和月度复盘在每日记录、基础图表和每日复盘稳定后再进入后续 Step。

AI 输入原则：

```text
程序统计结果
+ 关键事件摘要
+ 情绪/任务/习惯数据
+ 少量必要原文片段
```

AI 不默认读取全部原始日记。每日复盘允许少量事件原文进入上下文，但必须经过发送前判定。

每日复盘原文发送规则：

- 只有事件记录可以进入原文候选；灵感默认只进入状态或数量摘要。
- 每条事件记录需要支持 AI 分析权限：不参与 AI 分析、仅摘要参与、允许原文参与。
- 默认权限为仅摘要参与。
- 发送前先由程序规则检查明显敏感内容，命中敏感规则时降级为摘要，不发送原文。
- 点击生成每日复盘后，先展示将发送给 AI 的内容预览，再由用户确认生成。
- 第一版最多发送 3-5 条事件原文，每条原文需要截断到合理长度，避免成本和隐私失控。

复盘结果需要保存到数据库，避免重复调用。

### 4.5 个人说明书

个人说明书是长期模块，第一版轻量实现。

内容：

- 当前人生阶段
- 当前主要目标
- 能力画像
- 情绪模式
- 高能量来源
- 常见内耗点
- 反复出现的问题
- 适合 Faye 的行动建议风格

更新方式：

- 第一版允许手动编辑。
- 后续由周复盘和月度复盘生成建议更新项，用户确认后写入。

### 4.6 场景工具箱

场景工具箱用于处理具体生活问题。

第一版候选工具：

- 情绪复盘
- 边界感练习
- 职场沟通复盘
- 关系复盘
- 决策复盘
- 压力整理
- 明日计划生成
- 周目标拆解

工具箱可以调用 AI，但必须由用户主动触发。

### 4.7 纪念日和礼物

从 life-notes skill 迁移的生活记录能力。

功能：

- 创建纪念日
- 设置提醒日期
- 记录礼物
- 关联纪念日和礼物
- 在临近纪念日时展示历史礼物和准备建议

第一版可以先做记录和列表，提醒能力可放到后续阶段。

## 5. 非 MVP 范围

第一版暂不做：

- 完整成长知识库
- Obsidian 全量同步
- 多用户公开社区
- 付费订阅
- 手机 App
- 复杂日历双向同步
- 自动从聊天记录中持续抽取任务

Obsidian 继续作为学习笔记主阵地。未来只预留 Markdown 导入、引用和导出能力。

## 6. 数据策略

### 6.1 主数据库

主数据库建议使用 Supabase Postgres。

原因：

- 适合结构化生活数据。
- 适合未来公开上线。
- 支持认证、权限、备份和查询。
- Supabase 有免费层，足够支撑自用 MVP 和早期验证；未来公开上线后再评估是否升级付费计划。

### 6.2 数据实体草案

第一版核心实体：

- User
- Task
- Habit
- HabitCheckin
- ScheduleItem
- LifeEntry
- EmotionTag
- InsightReport
- PersonalManual
- Anniversary
- GiftRecord
- ToolSession

具体字段在架构文档和数据库设计阶段细化。

## 7. AI 策略

### 7.1 接入方式

AI 通过后端 API 接入，不通过前端直连。

正式结构：

```text
前端页面
  ↓
后端 Route Handler / Server Action
  ↓
AI Provider Adapter
  ↓
OpenAI-compatible API / 国内模型 API / OpenAI API
  ↓
分析结果写入数据库
```

### 7.2 模型供应商策略

不绑定 OpenAI。

第一版采用 OpenAI-compatible provider adapter：

- 可以接低成本国内模型。
- 可以接免费或低价额度。
- 可以后续切换 OpenAI 高质量模型。
- 每种复盘场景可以选择不同模型。

模型使用建议：

- 每日复盘：低成本模型
- 周复盘：低成本或中等模型
- 月度复盘：中等模型，必要时切换更强模型
- 普通统计图表：不调用 AI

### 7.3 成本控制

- 手动点击或定时生成，不随页面打开重复生成。
- 复盘结果缓存到数据库。
- 同一日期、同一类型报告默认只生成一次。
- 支持重新生成，但需要明确操作。
- 输入内容先由程序压缩为结构化摘要。

### 7.4 隐私原则

- API key 只存在服务端环境变量。
- 前端不暴露任何模型密钥。
- 默认不发送完整原始日记。
- 敏感记录未来可增加“不参与 AI 分析”开关。

## 9. 基础设计要求

视觉方向：

- 安静、清晰、实用。
- 更像长期使用的个人工作台，不像营销落地页。
- 信息密度适中，优先保证每天打开时能快速记录和查看。

配色方向：

- 低饱和度、温暖、莫兰迪色系。
- 主色可以从紫色、绿色、蓝色中组合。
- 避免高饱和霓虹色、强烈渐变和过度装饰。
- 图表颜色要柔和，但不同数据类型需要能清楚区分。

布局方向：

- 桌面端优先 dashboard 布局。
- 移动端保证基础记录流程顺畅。
- 不做大面积营销 hero。
- 不使用纯装饰性背景元素抢占注意力。

## 10. 成功标准

第一版成功标准：

- Faye 可以每天打开工作台完成任务、打卡、随手记录。
- 晚上可以一键生成今日复盘。
- 每周可以生成一份可读、有行动建议的周复盘。
- 图表能反映任务、习惯、情绪和记录趋势。
- AI 调用次数可控，不依赖昂贵模型完成普通功能。
- 数据结构能支撑未来迁移、扩展和上线。
